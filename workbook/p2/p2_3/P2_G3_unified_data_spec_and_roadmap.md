# P2-G3 — 통일 데이터 명세 & 모델링 이전 최종 데이터셋 청사진

**목적**: P2-G4(모델링) 착수 전, `p3_1.ipynb`(입결·A비율·취업진학·임금·자격증 트랙)와 `p3_2.ipynb`(GOMS 노동시장 트랙) 두 개의 서로 다른 에이전트가 관리하는 노트북을 실제로 검증한 뒤, 하나로 합칠 수 있는 통일 데이터 명세와 각 에이전트에게 하달할 구체적 규칙·목표·실행 프롬프트를 확정한다.

**검증 방법**: 두 노트북(총 51개 코드 셀)을 전량 읽고, 6개 입결/임금/자격증 원천 CSV와 GOMS 39개 정규화 결과를 Python으로 직접 로드해 실측 검증했다 (건수, 결측률, 유니크값, 매칭률 전부 실제 수치). 이전 에이전트가 제시한 의견은 **틀린 부분을 포함하고 있었다** — 아래 "검증 결과"에서 CONFIRMED/PARTIAL/REFUTED로 명시한다.

## 1. 검증 결과 요약

### 1.1 p3_1 트랙 (입결/A비율/취업진학/임금/자격증)

`p3_1.ipynb`(9셀)는 **병합을 전혀 시도하지 않았다** — 6개 원천 파일 각각을 독립적으로 프로파일링(스키마 요약·결측률·target 후보·describe)만 수행했다. 이전 에이전트가 제시한 병합 전략(4단계 매칭 ladder, 8개 컬럼 역할 분류, 3개 target mart 등)은 **이 노트북 안에서 실행·검증된 바가 없는 순수 제안**이었다.

| 이전 에이전트 주장 | 판정 | 실측 근거 |
|---|---|---|
| "메인 파일에 성적/취업/진학 컬럼이 있다" (묵시적 전제) | **REFUTED** | `P2_G2_메인_입결_A_취업진학.CSV`(15,727×120)에는 A/CD/F비율·취업률·진학률 컬럼이 **단 하나도 없다**. 순수 정원·지원자·입학자·재학생·졸업자·교원 카운트 마스터다. 파일명과 실제 스키마가 불일치하므로 **개명 필요** (`dept_headcount_master_2024` 등). 실제 target은 전부 `P2_G2_정시입결.csv`(10,242×17)에 있다. |
| "학위과정/학제 필터링이 필요하다" | **CONFIRMED** | 학위과정 3종(대학과정7,494 / 대학원과정7,467 / 전문대학과정766), 학제 10종 실측 확인. |
| "학교-학과 키가 중복되어 즉시 병합 불가하다" | **PARTIAL — 원인 정정** | `학교명+학과명` 중복은 1.45%(228행)로 작지만, 원인은 "동일 학과가 여러 행에 반복"이 아니라 **캠퍼스(본분교) 차이**였다(강원대 춘천/삼척, 가천대 성남/인천 등). 학과코드도 캠퍼스 간 거의 동일해 구분 불가. → 키에 **본분교(또는 시군구)를 반드시 포함**해야 하며, 이전 에이전트가 제안한 "count 컬럼은 sum 집계" 정책을 그대로 적용하면 서로 다른 캠퍼스를 합쳐버리는 오류가 난다. |
| "정시입결-메인 완전일치율이 낮아 정규화·유사도 매칭이 필요하다" | **CONFIRMED** | 학교명 완전일치는 48.5%, (학교명,학과명) 쌍 완전일치는 42.3%뿐. 대계열("공학계열") vs 임금분류 계열("공학") 접미사 차이도 확인. |
| "직무별_자격증매핑은 학과 단위 직접 조인 불가, bridge 필요" | **CONFIRMED (강함)** | `join_now` 컬럼이 24건 **전부 False**. review_needed 66.7%, confidence low/exclude/unknown 합계 25%. 즉시 조인 가능 행이 실측상 0건. |
| (신규 발견) 정시입결의 `학과_계열`은 표준 대계열이 아니다 | **추가 확인** | 4,195개 고유값으로 학과명에 가까운 세부 라벨이다(예: 경영120건, 사회복지114건). 7종 표준 대계열로 쓰려면 별도 crosswalk가 필요하다. |

### 1.2 p3_2 트랙 (GOMS 노동시장)

`p3_2.ipynb`(43셀)는 **내가 이미 만든 정규화 long-format 파일(`goms_distribution_long.csv`/`goms_continuous_long.csv`)을 전혀 쓰지 않고**, 원본 `.xls`(HTML)를 직접 재파싱해 `year__subgroup__측도` 형태의 **별도 wide-format 프로파일**을 만들었다. 원본 데이터 자체는 두 위치(`p2_2/data/goms_subject_crawl`, `p2_2/final/final_data`) 모두 byte-identical함을 `diff -rq`로 확인했으나, **p3_2.ipynb의 산출물은 모델링 입력이 아니라 "육안 검증용 wide EDA"로만 써야 한다** — 실제 mart는 반드시 이미 검증된 long 파일에서 만들어야 스키마 혼선이 없다.

| 이전 에이전트 주장 | 판정 | 실측 근거 |
|---|---|---|
| "39개 파일을 초대형 wide로 직접 병합하면 안 된다" | **CONFIRMED** | 이미 long-format 정규화가 완료돼 있고(29,160행+2,230행, 파싱 실패 0, 중복 0), 이걸 다시 wide 병합하는 것은 회귀다. |
| "주제마다 dimension_value/subgroup 축 의미가 다르다" | **CONFIRMED (구체 사례로)** | GOMS_003: dimension_value=전공계열, subgroup=경제활동상태. GOMS_006: dimension_value=산업, subgroup=전공계열 (축이 뒤바뀜). GOMS_009: dimension_value=직업, subgroup=학교유형. |
| "GOMS_001~003=전체졸업자, GOMS_004 이후=취업자 조건부" | **PARTIAL — 보완 필요** | topic_name 접두어("취업자의") 기준으로는 GOMS_004~021(18개, 그룹3)만 취업자 모집단이다. 그러나 GOMS_022~039(근로소득9+근로시간9, 그룹4·5)는 접두어가 없지만 **논리적으로 취업자만 응답 가능한 항목**(무직자는 소득·근로시간이 없음)이라 텍스트 접두어만으로 모집단을 나누면 이 18개가 "전체졸업자"로 오분류된다. → `population_scope`는 topic_name 텍스트가 아니라 **topic_group(그룹번호)** 기준으로 규칙화해야 한다. |
| "전공계열 표기가 일관된다" | **CONFIRMED** | 인문/사회/교육/공학/자연/의약/예체능 8개(전체 포함) 문자열이 모든 topic에서 byte 단위로 완전히 동일 (`repr()` 검사, 공백/BOM 차이 없음). |
| "직업분류 2016 이전/2017 이후를 분리 유지해야 한다" | **CONFIRMED** | pre_2017(2007~2016)과 post_2017(2017~2019) 연도 범위가 서로 겹치지 않고 정확히 분리됨을 확인. |
| "표준편차는 사이트 버그로 취득 불가하다" | **CONFIRMED (기 문서화)** | `qa/final_qa_summary.csv`에 근거 포함, 재확인만 함. |

## 2. 현재 데이터 지형

3개의 독립적 소스 패밀리가 있고, **공통 브리지 키가 하나로 수렴한다**: `major_group_7` (전공계열 7종: 인문/사회/교육/공학/자연/의약/예체능).

```
┌─────────────────────────────┐   ┌──────────────────────────────┐   ┌───────────────────────────────┐
│ A. 대학·학과 코어 (2024 단면) │   │ B. GOMS 노동시장 (2007~2019)  │   │ C. 임금·자격증 참조테이블      │
│  담당: p3_1 에이전트          │   │  담당: p3_2 에이전트           │   │  담당: p3_1 에이전트           │
│                              │   │                               │   │                               │
│ 메인(정원/재적/졸업/교원)     │   │ goms_distribution_long.csv    │   │ 임금분류_학부대학원 (14행)     │
│  grain=학교×학과×학위과정     │   │  (범주형 21종, 29,160행)      │   │  grain=대학원여부×계열(7)      │
│  15,727행                    │   │ goms_continuous_long.csv      │   │ 직무별_자격증매핑 (24행,       │
│ 정시입결(A/CD/F비율,취업률,   │   │  (연속형 18종, 2,230행)       │   │  join_now 전부 False → bridge  │
│  진학률) grain=학교×학과_전공  │   │  grain=전공계열(7) × year     │   │  전용, 직접 조인 금지)         │
│  10,242행                    │   │  (주제별로 축 의미 상이)       │   │                               │
└──────────────┬───────────────┘   └───────────────┬───────────────┘   └───────────────┬───────────────┘
               │  school+dept 정규화·유사도 매칭          │  major_group_7 (byte-identical, 안전)         │  계열 접미사 제거
               │  (현재 42~48% 매칭률, 개선 필요)          │                                                │  (즉시 100% 매칭)
               └───────────────────────┬───────────────────────────────────────────────────────────────────┘
                                        ▼
                          공통 키: major_group_7 (+ year, 필요시 학교/학과)
```

핵심 관찰: **GOMS(B)와 임금분류(C)는 전공계열 7종 단위로만 존재**하므로 100% 매칭이 보장된다. 반면 **정시입결의 `학과_계열`(4,195종)은 7종 표준 대계열이 아니다** — 메인 파일의 `대계열`(계열 접미사만 떼면 7종과 일치)에서 상속받거나, 별도 crosswalk를 만들어야 한다. 이것이 이번 로드맵에서 가장 중요한 미해결 조인 지점이다 (§9 참고).

## 4. 핵심 설계 결정

1. **39개 파일을 wide 하나로 합치지 않는다** — long fact → mart 2단계 유지(§1.2에서 재확인된 원칙).
2. **major_group_7이 유일한 만능 키가 아니다** — 정시입결의 `학과_계열`(4,195종)은 표준 대계열이 아니므로, ③`dept_master_2024`가 만들어진 뒤에만 메인의 `대계열`을 상속받아 `major_group_7`을 채울 수 있다. 즉 **⑥ GOMS 조인은 반드시 ③이 완성된 뒤에 진행**해야 한다(순서 의존성).
3. **campus(본분교)를 키에서 빼지 않는다** — 이전 에이전트의 "동일 학교-학과는 sum 집계" 정책은 서로 다른 캠퍼스를 합쳐버리는 오류를 유발하므로 폐기하고, `학교명+학과명+본분교`를 최소 키로 강제한다.
4. **population_scope는 텍스트가 아니라 topic_group 코드로 규칙화한다** — GOMS_022~039도 사실상 취업자 조건부 데이터로 플래그한다.
5. **입결_프록시는 주 타겟에서 배제** — 결측 63.5%로 모델 품질을 해칠 위험이 크다.
6. **자격증-직무 매핑은 이번 라운드에서 조인하지 않는다** — `join_now=False` 24/24 실측 확정, 별도 보관만 한다.
7. **target은 아직 고정하지 않는다** — 데이터 확정이 목적이며, target 선택은 P2-G4 연구질문 확정 후 진행한다(이전 에이전트의 이 결론 자체는 타당하여 유지).

## 5. p3_1 담당 에이전트 — 정책·규칙·목표

**소유 원본**: `P2_G2_메인_입결_A_취업진학.CSV`, `P2_G2_정시입결.csv`, `P2_G2_임금분류_학부대학원.CSV`(+사분위기준/컬럼설명), `P2_G2_직무별_자격증매핑.CSV`

### 5.1 목표
1. 메인 파일을 리네임 검토 후 `dept_headcount_master_2024`로 정제(①), 정시입결을 `dept_outcomes_2024`로 정제(②).

5. 임금분류 3종 파일을 정제해 ④를 만든다(계열 접미사 정규화만 하면 즉시 100% 매칭 가능함을 활용).

### 5.2 구체 규칙
- 카운트 컬럼(지원자·입학자·재학생·졸업자·교원 등)은 **캠퍼스가 확정된 키에서만** 필요 시 합산한다. 키 확정 전 임의 sum 금지.
- 비율 컬럼은 절대 평균하지 않는다 — 분자/분모가 있으면 재계산, 없으면 그대로 보존.
- `입결_프록시`는 `dept_outcomes_2024`에 유지하되 컬럼 설명에 "결측 63.5%, 보조 feature 전용"이라고 명시.
- 매칭률 목표: 현재 (학교명,학과명) 완전일치 42.3% → 정규화+유사도 매칭으로 **70% 이상**을 1차 목표로 한다. 70% 미만이면 이유(캠퍼스 표기, 학과 통폐합 등)를 `data/qa/dept_match_gap_report.csv`류로 남긴다.

## 6. p3_2 담당 에이전트 — 정책·규칙·목표

**소유 산출물**: ⑥⑦
**소유 원본**: `normalized/goms_distribution_long.csv`, `normalized/goms_continuous_long.csv` (반드시 이 두 파일에서 시작 — 원본 `.xls` 재파싱 금지)

### 6.1 목표
1. long 파일에서 `major_group_7`이 dimension_value 또는 subgroup으로 등장하는 topic만 선별해(GOMS_003,006,011,012,015,018,021,024,033 등) `year×major_group_7` grain으로 pivot한다.
2. 산업/직업처럼 카테고리 20개 이상인 항목은 **direct wide pivot 대신** top1/top3 share, HHI(`Σp²`), entropy(`-Σp·log p`) 등 요약 피처로 압축한다.
3. 직업 관련 피처는 `_pre2017`/`_post2017` 접미사로 분리 유지하고, 두 시기를 값 단위로 이어붙이지 않는다.
4. 연속형(소득·근로시간) topic은 이미 정제된 mean 값만 사용하고(표준편차 없음), `hourly_income_proxy` 파생 시 반드시 `_proxy` 접미사를 붙인다.
5. `population_scope`는 topic_group 코드 기준으로 재규정한다: 그룹2(경제활동상태)=전체졸업자, 그룹3(현재일자리)·그룹4(근로소득)·그룹5(근로시간)=취업자 조건부.
6. (우선순위 낮음) 성별·학교유형 mart(⑦)는 시간 여유가 있을 때만 생성한다.

### 6.2 구체 규칙
- `전체` 행(dimension_value 또는 subgroup)은 삭제하지 말고 `is_total` 플래그로 표시하거나 별도 national-benchmark 테이블로 분리한다.
- share 계열 값은 GOMS_003처럼 **조건부 확률의 분모가 바뀌는 경우**를 반드시 구분한다(예: `P(전공|경제활동상태)` vs `P(산업|전공)`는 의미가 다르다 — 전공별 경제활동률이 필요하면 frequency로 재계산, share를 그대로 쓰지 않는다).
- 최종 mart의 컬럼 수는 30~45개 내외(compact)로 관리하고, 필요 시 전체 상세본(`*_full`)을 별도로 남긴다.
- 참조 연도 정책을 명시적으로 정한다: 2024년 학과 데이터와 조인 시 GOMS 최신연도(2019)를 쓸지, 다년 평균을 쓸지 **하나를 선택하고 문서화**한다(임의 혼용 금지).

## 7. 공통 규칙 (양쪽 에이전트 공통 적용)

1. **grain 선언 의무**: 산출물마다 "한 행 = ___" 문장을 코드/문서에 명시한다.
2. **long끼리 직접 merge 금지**: 카테고리 수가 다른 long table을 그대로 merge하면 Cartesian 행폭증이 난다. 반드시 pivot/요약 후 병합한다.
3. **`validate` 파라미터 필수**: 모든 `pd.merge`에 `validate="one_to_one"`/`"many_to_one"` 등을 명시하고, 실패 시 예외를 내야 한다(조용한 병합 금지).
4. **dtype 표준**: 식별자=`string`, 분류/전략=`category`, 개수=`Int64`(nullable), 비율=`float64`(0~100 스케일로 통일, 0~1과 혼용 금지), 플래그=`bool`.
5. **품질 컬럼 보존**: 모든 최종 산출물에 최소 하나의 provenance/quality 컬럼(`source_*`, `match_*`, `missing_feature_count` 등)을 남긴다.
6. **원본 불변**: `P2_G2_*.CSV`, `raw_downloads/*.xls`는 절대 수정하지 않는다. 정제본은 새 파일로만 저장한다.
7. **target 강제 금지**: 이번 라운드 산출물에 특정 target을 고정하지 않는다. `dept_outcomes_2024`에 후보 target들을 나열만 하고 선택은 P2-G4로 넘긴다.

## 9. 남은 리스크·미결정 사항

1. **최대 리스크 — `major_group_7` 상속 경로**: 정시입결의 `학과_계열`(4,195종)은 표준 대계열이 아니므로, `dept_outcomes_2024` 단독으로는 `major_group_7`을 채울 수 없다. 현재 계획은 `dept_master_2024`(메인+정시입결 병합 성공분)를 통해서만 `major_group_7`을 상속받는 것인데, 매칭률이 42~48%에 머물면 GOMS 조인 커버리지도 그만큼만 확보된다. **대안**: `학과_계열`(4,195종) → `major_group_7`(7종) 직접 crosswalk를 키워드 규칙(간호→의약, 컴퓨터공→공학 등)으로 별도 구축하면 매칭 실패 건에도 100%에 가까운 GOMS 조인이 가능해진다. 이 crosswalk는 **아직 존재하지 않으며**(`P2_학과별_A비율_대학아님.csv`, `P2_admission_proxy_v3_by_department.csv` 확인 결과 없음), p3_1 에이전트가 §5 목표에 추가로 만들지, 별도 3번째 에이전트가 만들지는 **사용자 결정이 필요**하다.
2. **GOMS 참조연도 정책 미확정**: 2024년 학과 데이터에 2007~2019년 GOMS 중 어느 연도(들)를 붙일지 아직 규칙이 없다 (§6 목표 5에서 p3_2가 확정하도록 위임했으나, 이는 전체 파이프라인의 해석에 영향을 주므로 최종 승인은 사용자가 해야 한다).
3. **`dept_master_2024`의 최종 행수 범위(4,300~10,000)**: 매칭 개선 결과에 따라 달라지며, 목표(70%) 미달 시 모델링 표본 크기 자체가 재검토 대상이 된다.
4. **target 미확정은 의도된 것**: §4-7에 따라 이번 단계는 target을 고정하지 않는다. P2-G4에서 연구질문(성적↔A비율, A비율↔취업, A비율↔진학 중 택)을 정한 뒤 `dept_outcomes_2024`에서 y를 선택해야 한다.
